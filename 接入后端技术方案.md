# Berg PC 案件管理平台 — 接入后端技术方案

> 现状:当前平台是**纯前端**应用,全部数据存在浏览器 `localStorage`(键 `berg_platform_v1`),通过 `app/db.js` 的 `window.DB` 接口读写。优点是能离线演示、刷新不丢;局限是**数据只存在本机、无法多人共用、无真实权限校验**。本方案说明如何升级为多人协作、数据存云端的真实系统。

---

## 1. 目标与差距

| 能力 | 现在(前端 localStorage) | 接后端之后 |
|---|---|---|
| 数据存储 | 本机浏览器 | 云端数据库,多设备共享 |
| 多用户协作 | 不可(各自本机) | 律师/客户/管理员实时共用同一份数据 |
| 登录鉴权 | 明文比对、无 token | 加盐哈希密码 + JWT/会话,服务端校验 |
| 权限隔离 | 前端过滤(可被绕过) | 服务端强制(行级权限) |
| 文件/证据 | 仅登记元数据 | 真实文件上传到对象存储 |
| 审计日志 | 本机 | 服务端不可篡改日志 |

---

## 2. 推荐架构(两条路线)

**路线 A — 快速上线(推荐起步):Supabase / Firebase**
- 托管式 Postgres(Supabase)或 Firestore(Firebase),自带鉴权、行级安全(RLS)、对象存储、实时订阅。
- 前端几乎不用改结构:把 `db.js` 的每个方法从「读写 localStorage」改成「调用 Supabase SDK」。
- 适合:几周内上线、团队规模中小、不想自管服务器。

**路线 B — 自建后端(完全可控):Node + Postgres**
- `Node.js (Express/Fastify)` + `PostgreSQL` + `Prisma`(ORM)+ `S3 兼容对象存储`(AWS S3 / 阿里云 OSS)。
- 前端通过 REST 或 tRPC 调用。
- 适合:需自主合规、私有部署、与现有系统集成。

---

## 3. 数据模型(Postgres 表结构草案)

直接映射现有 `db.js` 的实体:

```sql
users        (id, role[admin|lawyer|client], name, email UNIQUE, password_hash, title, created_at)
clients      (id, user_id FK, lawyer_id FK, name, email, phone, address, dob, occupation,
              id_type, id_number, status, matter_type, amount_lost, currency, date_of_loss,
              platform, funds_method, jurisdiction, overseas, narrative, intake_complete,
              created_by, created_at, updated_at)
wallets      (id, client_id FK, addr, role, label, balance, usd)
strategies   (client_id FK, strategy_id)              -- 多对多
documents    (id, client_id FK, name, type, size, status, storage_key, uploaded_by, uploaded_at)
timeline     (id, client_id FK, icon, text, by, date)
messages     (id, client_id FK, from_id FK, text, ts, read)
notes        (id, client_id FK, author, text, ts)     -- 仅 lawyer/admin 可见
activity     (id, actor, action, target, ts)
```

---

## 4. 权限规则(服务端强制)

- **client**:只能读/写 `client.user_id == 自己` 的那一条;笔记(notes)对其完全不可见。
- **lawyer**:只能访问 `client.lawyer_id == 自己` 的客户;可改资料、钱包、策略、证据状态、时间线、笔记。
- **admin**:全部可见;独占用户管理与「客户分配」(改 `clients.lawyer_id`)。

> Supabase 用 RLS 策略一条 SQL 即可实现;自建后端在中间件里按 `req.user.role` 过滤。**关键:权限必须在服务端判断,前端过滤只是 UI 便利。**

---

## 5. 迁移步骤(最小改动)

1. **保留** `db.js` 的方法签名(`login`, `clientsFor`, `updateClient`, `sendMessage`…)不变 —— 它已经是一层清晰的数据访问接口。
2. 新建 `api.js`,用同样的方法名,内部改为 `await fetch(...)` / Supabase SDK 调用。
3. 把组件里的 `window.DB.xxx()` 改为 `await window.API.xxx()`(加 `async/await`)。
4. 鉴权:登录返回 token,存 `httpOnly` cookie;每次请求带上。
5. 证据上传:`<input type=file>` → 预签名 URL 直传对象存储 → 回存 `storage_key`。
6. 实时:Supabase Realtime / WebSocket 订阅 `messages`,实现留言即时到达。

因为数据访问已经收口在 `DB` 这一层,**业务组件基本不用重写**,这是当前架构最省力的地方。

---

## 6. 工作量与时间(粗估)

| 阶段 | 路线 A(Supabase) | 路线 B(自建) |
|---|---|---|
| 建库 + 鉴权 + RLS | 3–5 天 | 1–2 周 |
| db.js → api.js 改造 | 3–4 天 | 3–5 天 |
| 文件上传 + 实时留言 | 2–3 天 | 1 周 |
| 测试 + 部署 | 2–3 天 | 1 周 |
| **合计** | **~2–3 周** | **~4–6 周** |

---

## 7. 合规提醒(律所场景)

- 涉及当事人敏感信息与证据,需:传输加密(HTTPS)、静态加密、访问审计、定期备份、数据驻留地选择(美国/本地)。
- 律师–客户保密:笔记与留言需严格按角色隔离;建议保留完整不可篡改的审计日志(本方案 `activity` 表已预留)。
- 如面向美国客户,考虑 SOC 2 / 数据处理协议(DPA)。
